# DockerHost

Основная рабочая лошадка сервисов — VM (Ubuntu) на гипервизоре PVE, `192.168.50.30` (SERVICES). Внутри крутится Docker-стек из десятков контейнеров: медиасервер, arr-стек, фотохранилище, видеонаблюдение и вспомогательные сервисы. Публикуются наружу через Traefik.

Развёрнута как VM (не LXC) сознательно: нужен проброс физических дисков под ZFS-пулы и iGPU для аппаратного транскодинга — то, что чисто делается только на уровне VM.

## 1. Виртуализация и железо

VM на PVE с проброшенными ресурсами:

- **iGPU passthrough** — интегрированная графика Intel проброшена в VM для аппаратного транскодинга (Jellyfin) и детекции (Frigate). Без него транскод и анализ видео легли бы на CPU.
- **Passthrough физических дисков под ZFS** — диски проброшены в VM целиком, ZFS-пулы собираются и импортируются **внутри** VM, а не на гипервизоре. На каждом проброшенном диске в конфиге VM выставлен флаг `backup=0`, чтобы PBS не пытался бэкапить многотерабайтные пулы (см. `05-backup.md`). PBS бэкапит только системный диск VM.

Метаданные ZFS живут на самих дисках, поэтому при пересоздании VM пулы поднимаются через `zpool import` без потери данных.

## 2. ZFS-датасеты

Внутри VM три пула под разные задачи:

| Пул | Назначение | Бэкап |
| :--- | :--- | :--- |
| `zdata` (raidz1) | AppData контейнеров, конфиги, compose-файлы, БД сервисов | restic (Backrest) |
| `zmedia` | медиатека (фильмы, сериалы, музыка) для Jellyfin | не бэкапится (перекачиваемо) |
| `zfrigate` | записи камер Frigate | не бэкапится (циклическая перезапись) |

Критичное состояние (`zdata`: AppData, конфиги, БД) покрыто restic. Медиатека и записи камер — большие и восстановимые/эфемерные, их бэкап нецелесообразен.

## 3. Docker-стек

Стек управляется через Docker Compose, AppData и compose-файлы лежат на `zdata`. Основные группы сервисов:

- **Медиа**: Jellyfin (медиасервер, аппаратный транскод через iGPU), Jellyseerr (запросы контента).
- **arr-стек**: Sonarr, Radarr (управление медиатекой), qBittorrent (загрузка) и сопутствующие индексеры.
- **Фото**: Immich (фотохранилище с ML-функциями).
- **Видеонаблюдение**: Frigate (NVR с детекцией объектов на iGPU) — см. раздел 5.
- **Вспомогательные**: Backrest (управление restic-бэкапами через веб-UI), Homepage (дашборд), и прочие утилиты.

Сервисы, которые публикуются наружу, проксируются через Traefik по своим доменам `*.kvasok.xyz` (см. `07-traefik.md`), обычно под цепочкой `chain-external` + Authelia, а Jellyfin — доступен телевизорам через media-доступ (см. `07-traefik.md`, раздел про `allow-media-ips`).

## 4. Сеть

Внутри VM контейнеры общаются через Docker-сети (bridge). Наружу VM имеет один адрес `192.168.50.30` в SERVICES.

**Входящий трафик** — только через Traefik (`192.168.40.11`): проксирование сервисов и виджет Homepage (Traefik API). nftables на VM пускает порты сервисов только с Traefik (см. раздел 7).

**Исходящий гео-обход для arr-стека** — часть контейнеров (arr-стек, qBittorrent) ходит в интернет через Xray-прокси: в их окружении прописан `HTTP_PROXY`/`HTTPS_PROXY` на HTTP-порт Xray (`http://192.168.20.12:10809`). Это направляет их исходящий трафик через гео-обход (индексеры, недоступные напрямую через провайдера), тогда как остальной трафик VM идёт напрямую. Детали Xray — в `09-xray.md`.

## 5. Frigate и камеры

Frigate — NVR в Docker, работает в **pull-режиме**: сам инициирует подключение к камерам за потоком, камеры соединения не инициируют. Забирает RTSP-поток (порт 554) и управляет через ONVIF (порт 2020) с камер в CCTV (`192.168.70.11`, `192.168.70.12`).

Поскольку Frigate (в SERVICES) инициирует соединение к камерам (в CCTV), на OPNsense заведено разрешающее правило `192.168.50.30 → cctv_hosts:cctv_ports` выше блока приватных сетей (см. `03-network.md`). Сами камеры наружу порты не пробрасывают; их «облачный» доступ (Tapo) идёт отдельно через интернет из CCTV. Детекция объектов ускоряется iGPU. Записи пишутся на `zfrigate`.

## 6. Samba

На VM поднят Samba-сервер для доступа к медиатеке и данным с других устройств сети (файловые шары `zmedia` и части `zdata`). Конфиг `/etc/samba/` попадает в restic-бэкап. Доступ ограничен на уровне nftables и Samba `hosts allow`/`hosts deny`.

## 7. Файрвол (nftables)

nftables на VM обеспечивает внутрисегментную защиту (OPNsense фильтрует только межсегментный L3-трафик, внутри SERVICES хосты видят друг друга на L2 — см. `02-conventions.md`). `policy drop`, разрешены: порты проксируемых сервисов только с Traefik (`192.168.40.11`), Samba только с разрешённых источников, SSH из MGMT и VPN. Исходящие соединения (к Xray-прокси, к камерам, в интернет) разрешены.

## 8. Резервное копирование

Два уровня (см. `05-backup.md`):

- **PBS-снапшот** — только системный диск VM (проброшенные ZFS-диски исключены флагом `backup=0`).
- **restic через Backrest** — критичные данные с `zdata` (AppData, конфиги, compose-файлы, `/etc/samba/`, БД сервисов). Backrest — веб-UI-обёртка над restic в Docker, управляет расписанием и retention. Транспорт — rest-server на PBS (`192.168.10.15:8000`, репозиторий `dockerhost`, append-only). `zmedia` и `zfrigate` не бэкапятся.

БД сервисов, чувствительные к консистентности, снимаются корректным способом (дамп/остановка) перед включением в снапшот restic.

## 9. Зависимости

- **PVE** — гипервизор, проброс iGPU и ZFS-дисков.
- **Traefik (`192.168.40.11`)** — единственный источник входящих запросов к сервисам; публикация наружу.
- **Xray (`192.168.20.12`)** — HTTP-прокси гео-обхода для arr-стека и qBittorrent.
- **Камеры CCTV (`192.168.70.11/.12`)** — источники RTSP/ONVIF для Frigate.
- **PBS (`192.168.10.15`)** — PBS-снапшот системного диска и restic-репозиторий `dockerhost` (Backrest).
- **Authelia** — forward-auth для части публикуемых сервисов.
