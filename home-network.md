# Домашняя сеть

Документ описывает текущую сетевую инфраструктуру homelab: маршрутизацию,
VLAN-сегментацию, физическую коммутацию, беспроводную сеть и DNS.

---

## Общая архитектура

Ядром сети выступает **OPNsense**, работающий как единственный маршрутизатор,
firewall и DHCP/DNS-сервер. OPNsense запущен виртуальной машиной на мини-ПК
под управлением Proxmox VE. Весь трафик между сегментами и выход в интернет
проходят через него.

Кабель провайдера заходит напрямую в мини-ПK на физический интерфейс WAN.
Оба роутера Keenetic работают как беспроводные точки доступа (L2), маршрутизацию
не выполняют. Управляемый коммутатор TP-Link раздаёт VLAN тегированным транком
на все узлы.

Внешний доступ к публичным сервисам организован через VPS с WireGuard-туннелем
до внутреннего reverse-proxy (Traefik): интернет → VPS → WG-туннель → Traefik →
внутренний сервис.

---

## Схема VLAN

Сеть разделена на восемь VLAN. В каждом сегменте OPNsense выступает шлюзом с
адресом `.1`, раздаёт адреса через Kea DHCP и резолвит имена через Unbound.

| VLAN | Имя | Подсеть | Назначение |
| :--- | :--- | :--- | :--- |
| 1 | — | `192.168.1.0/24` | Не используется (native/legacy) |
| 10 | **MGMT** | `192.168.10.0/24` | OPNsense, коммутатор, Omada, оба Proxmox, PBS, точки доступа |
| 20 | **INFRA** | `192.168.20.0/24` | AmneziaWG, Xray |
| 30 | **TRUSTED** | `192.168.30.0/24` | Ноутбуки, ПК, телефоны, доверенные устройства |
| 40 | **DMZ** | `192.168.40.0/24` | Traefik — точка входа публичных сервисов |
| 50 | **SERVICES** | `192.168.50.0/24` | DockerHost VM, Vaultwarden, Authelia, прочие сервисы |
| 60 | **IOT** | `192.168.60.0/24` | Телевизоры, бытовая техника, умный дом |
| 70 | **CCTV** | `192.168.70.0/24` | IP-камеры |
| 80 | **GUEST** | `192.168.80.0/24` | Гостевой Wi-Fi |

Шлюз каждого сегмента — OPNsense (`192.168.<vlan>.1`). DHCP-пулы выделены в
диапазоне `.100–.200` каждой подсети.

## Гипервизоры

### PVE-Mini (Intel N150)

Мини-ПK с двумя сетевыми интерфейсами. Несёт сетевую плоскость дома.

**Сеть хоста:**

- `nic0` (`enxe43a6e8558c1`) → `vmbr0` — WAN. Мост без IP на хосте, чистый
  проброс физического интерфейса в WAN виртуальной машины OPNsense. В этот
  интерфейс заходит кабель провайдера.
- `nic1` (`enxe43a6e8558c2`) → `vmbr1` — LAN-транк. VLAN-aware мост, несёт все
  VLAN тегированными. Подключён в транк-порт коммутатора.
- `vmbr1.10` — управляющий адрес хоста `192.168.10.11` (MGMT), шлюз
  `192.168.10.1`. Через него же идёт default route хоста.

**Гости:**

- OPNsense (маршрутизатор/firewall): `net0` → `vmbr0` (WAN), `net1` → `vmbr1` (LAN-транк)
- Omada Controller
- AmneziaWG (LXC)
- Xray

### PVE-Main

Основной сервер. Подключён транком в коммутатор. Управляющий адрес хоста —
`192.168.10.12` (MGMT).

**Гости (с назначением VLAN на виртуальных интерфейсах):**

- DockerHost VM → VLAN 50 (SERVICES), `192.168.50.30`, проброс iGPU
- Traefik → VLAN 40 (DMZ), `192.168.40.11`
- Vaultwarden → VLAN 50 (SERVICES), `192.168.50.11`
- Прочие сервисы согласно схеме VLAN

## OPNsense

Виртуальная машина на PVE-Mini. Единственный маршрутизатор и firewall сети.

**Интерфейсы:**

- WAN (`vtnet0`) — подключён к `vmbr0` (физически `nic0`), получает белый IP от
  провайдера по DHCP. На WAN включены Block bogon networks и Block private
  networks.
- LAN-транк (`vtnet1`) — подключён к `vmbr1`, несёт восемь VLAN-интерфейсов
  (теги 10–80). На каждом VLAN OPNsense — шлюз `.1`.

**Сервисы:**

- **Kea DHCP** — на всех VLAN, пулы `.100–.200`.
- **Unbound DNS** — слушает на всех VLAN-интерфейсах. Реализует split-horizon:
  все `*.kvasok.xyz` резолвятся в `192.168.40.11` (Traefik) через Host Override.

**Firewall (текущее состояние):**

- **Floating:** форсирование DNS — разрешение DNS-запросов только к Unbound
  (порт 53 → This Firewall) и блокировка любого внешнего DNS (порт 53 → any).
- **Per-interface** на каждом VLAN: anti-lockout (доступ к управлению из своего
  сегмента) → правило «Block access to private networks» (в текущий момент
  отключено) → allow any.

На текущий момент межсегментный доступ открыт (правило блокировки приватных
сетей отключено на сегментах). Гранулярная изоляция между VLAN настраивается
отдельно.

## Коммутатор

**TP-Link SG2008P**, управляется через Omada Controller (LXC). Управляющий адрес
`192.168.10.21` (MGMT).

Раскладка портов:

| Порт | Назначение | Режим |
| :--- | :--- | :--- |
| 1 | Точка доступа Keenetic Giga | Trunk (native MGMT/10, tagged 30/60/80) |
| 5 | PVE-Main | Trunk |
| 6 | PBS | Access MGMT/10 |
| 8 | Uplink к PVE-Mini (`nic1`) | Trunk |

Транк-порты несут native-трафик как MGMT (VLAN 10) и тегированные
пользовательские VLAN.

## Беспроводная сеть

Два роутера Keenetic работают как **независимые точки доступа** (режим
«Ретранслятор», L2-коммутация без маршрутизации). Каждый вещает одинаковые SSID,
привязанные к VLAN-сегментам; устройства переключаются между точками по силе
сигнала.

- **Keenetic Giga (KN-1012)** — точка доступа, `192.168.10.25` (MGMT).
  Подключена транком в порт 1 коммутатора (native MGMT/10, tagged 30/60/80).
- **Keenetic Speedster (KN-3012)** — точка доступа, `192.168.10.26` (MGMT).
  Подключена по кабелю к Giga (транк с тегами нужных VLAN). Порт 1 Speedster
  настроен как access в VLAN 30 (TRUSTED) — в него проводом подключён ПК.

**Привязка SSID к VLAN:**

- Домашний SSID → TRUSTED (VLAN 30)
- SSID для телевизоров → IOT (VLAN 60)

DHCP на самих точках не поднимается — адреса всем клиентам раздаёт OPNsense.
Точки только мостят «тег VLAN ↔ SSID». Management-плоскость точек — в native
VLAN (MGMT).

## DNS

Резолвинг для всей сети выполняет **Unbound на OPNsense**. Клиенты в каждом VLAN
получают адрес шлюза своего сегмента (`192.168.<vlan>.1`) как DNS-сервер.

Split-horizon: запись Host Override `*.kvasok.xyz → 192.168.40.11` направляет все
локальные обращения к публичным доменам на внутренний Traefik. Устройства внутри
сети обращаются к сервисам напрямую по локальному адресу reverse-proxy, не
выходя в интернет.

Форсирование DNS на firewall гарантирует, что запросы идут только через Unbound:
внешний DNS (порт 53) заблокирован, обход резолвера невозможен.

## Внешний доступ

Публичные сервисы доступны из интернета через **VPS**, выступающий точкой входа.
Между VPS и внутренним Traefik поднят исходящий **WireGuard-туннель**. Порты на
домашнем роутере наружу не пробрасываются — соединение инициируется изнутри.

Путь внешнего запроса:

```
Интернет → VPS (публичный вход) → WireGuard-туннель → Traefik (192.168.40.11, DMZ)
        → внутренний сервис (SERVICES / прочие VLAN)
```

Traefik терминирует TLS и маршрутизирует запросы по доменам на соответствующие
бэкенды.

---

## Удалённый доступ (VPN)

**AmneziaWG** (INFRA, `192.168.20.11`) обеспечивает обфусцированный VPN-доступ в
сеть извне. VPN-подсеть — `10.8.0.0/24`, трафик клиентов маршрутизируется в
нужные сегменты через OPNsense.

---

## Поток трафика (сводно)

- **Выход в интернет:** клиент → шлюз своего VLAN (OPNsense) → WAN → провайдер.
- **Между сегментами:** клиент → OPNsense (маршрутизация + firewall) → целевой VLAN.
- **К локальным сервисам по домену:** клиент → Unbound (резолв в Traefik) →
  Traefik (DMZ) → бэкенд.
- **Извне к публичным сервисам:** интернет → VPS → WG → Traefik → бэкенд.
- **Извне в сеть (VPN):** клиент → AmneziaWG (INFRA) → OPNsense → целевой сегмент.
